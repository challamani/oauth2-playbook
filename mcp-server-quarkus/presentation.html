Problem Statement

AI Agents Are Taking Real-World Actions

The New Reality

  • AI agents don't just answer questions — they call APIs, book seats, send emails, trigger deployments
  • An agent without identity is an unauthenticated bot with your backend credentials
  • Every action should be traceable to a real user
  • Different users must get different capabilities
❓ Critical question:
Who is the AI agent acting on behalf of? And what is it allowed to do?

What Can Go Wrong

🔓 No auth: Any caller can invoke MCP tools — no identity, no limits

🔑 Shared secrets: All users share the same credentials — one breach = all exposed

📡 Token forwarding: The raw agent token reaches your database API — wrong audience, wrong scopes

🏔️ Over-privileged: Agents get more access than their user should have

MCP Explained

Model Context Protocol — The Glue Layer

What is MCP?

MCP (Model Context Protocol) is an open protocol that lets AI models discover and call tools hosted on external servers — think "USB-C for AI".

  • Transport: HTTP/SSE or STDIO
  • Tools: Functions the AI can call
  • Resources: Data the AI can read
  • Prompts: Reusable prompt templates
/.well-known/oauth-authorization-server
MCP servers advertise their OAuth endpoints here — clients auto-configure.

MCP Request Flow

🤖 AI Model (GitHub Copilot)
↓ JSON-RPC over HTTP + Access Token
🔧 MCP Server — tool definitions
↓ REST API call + Exchanged Token
📦 Resource Server — protected data
↓ Token validation
🔐 Keycloak — OAuth 2.1 Provider
Security

How to Protect MCP Servers

🛡️ Authentication

Every request to the MCP server must carry a valid JWT access token. Validate issuer, audience, expiry, and signature on every call.

No token → 401 Unauthorized

🔎 Scope Enforcement

Each MCP tool checks the required scope before executing. A token without the right scope gets a 403 — even if the user is authenticated.

mcp:reservations:write
mcp:events:read

🔄 Token Exchange

The MCP server exchanges the agent token for a narrow-scoped API token before calling the resource server. Raw agent tokens never reach the backend.

🎯 Audience Validation

Tokens are locked to their intended service via the aud claim. A token for the MCP server cannot be replayed against the resource server.

🔬 Least Privilege

Exchanged tokens only carry scopes that the user was originally granted. Token exchange cannot escalate privileges — ever.

🌐 Metadata Discovery

Expose /.well-known/oauth-authorization-server so AI clients auto-configure — no manual token setup in mcp.json.

OAuth 2.1

OAuth 2.1 — What Changed & Why It Matters

OAuth 2.0 → 2.1 Key Changes

  • PKCE is mandatory for all public clients (not optional)
  • Implicit flow is removed — too many attack vectors
  • Resource Owner Password flow is removed
  • Refresh tokens must be sender-constrained or rotated
  • Redirect URIs must be exact matches (no wildcards in prod)
OAuth 2.1 = OAuth 2.0 best practices, codified and mandatory

Two Grant Types in Focus Today

Grant Type 1
Authorization Code + PKCE

User authenticates in browser → AI agent gets token on behalf of user

Grant Type 2
Token Exchange (RFC 8693)

MCP server exchanges agent token → narrow API token for resource server

Grant Type 1

Authorization Code + PKCE

How PKCE Works

1
Client generates code_verifier

Random 43–128 char string, stored locally

2
Hash → code_challenge

BASE64URL(SHA256(code_verifier)) — sent in auth request

3
User logs in, code returned to client

Keycloak stores the challenge alongside the code

4
Token exchange: verifier sent

Keycloak validates SHA256(verifier) == challenge → issues token

When to Use & Advantages

AI Agents & Copilots SPA / Mobile Apps CLI Tools Desktop Agents
  • ✅ Secure for public clients — no client secret needed
  • ✅ Intercept protection — code useless without verifier
  • ✅ User-delegated — agent acts as the user
  • ✅ Auto-discoverable via /.well-known
  • ✅ GitHub Copilot supports it natively

Grant Type 2

Token Exchange — RFC 8693

The Two-Token Model

── Agent Token ──────────────────────────
aud:   mcp-agent
scope: mcp:events:read
       mcp:tickets:read
       mcp:reservations:write
sub:   manikanta

── Token Exchange Request ───────────────
grant_type:      urn:ietf:params:oauth:grant-type:token-exchange
subject_token:   <agent-token>
requested_scopes:api:inventory:reservations:write

── Exchanged Token ──────────────────────
aud:   mcp-inventory
scope: api:inventory:reservations:write
sub:   manikanta   ← identity preserved!

When to Use & Advantages

MCP → Resource Server Microservice Chains Cross-domain APIs
  • ✅ Trust boundary — MCP scopes ≠ API scopes
  • ✅ Raw agent token never reaches the database layer
  • ✅ Identity preserved — sub propagated
  • ✅ Cannot escalate — exchange is strictly downscoping
  • ✅ Audience-locked — tokens bound to correct service
🔒 Security guarantee: Even if the MCP server is compromised, it cannot exchange for scopes the user was never granted.

Reserving a Global Tech Conference via Copilot

🎬 Demo Flow

1
Configure MCP in Copilot

Add http://localhost:8085/mcp to mcp.json

2
Copilot discovers OAuth endpoint

Reads /.well-known/oauth-authorization-server — no manual token config

3
Browser login (PKCE flow)

User authenticates — Copilot receives a scoped JWT access token

4
"Reserve me a ticket to QCon London"

MCP tool → Token Exchange → Resource Server → Seat reserved ✅

🔄 Switching Users — See the 403

  1. Clear Keycloak SSO session (logout URL)
  2. Stop MCP server in Copilot Tools panel
  3. Start MCP server — browser login appears
  4. Log in as demo-readonly
  5. Ask Copilot to reserve a ticket
  6. → 403 Forbidden — scope not present at any layer
💡 The 403 is structurally impossible to bypass — enforced by Keycloak role mapping, MCP scope check, and token exchange validation simultaneously.
