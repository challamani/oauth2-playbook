Agenda

Securing AI Agents with OAuth 2.0 Grant Types

The core question: How does an AI Agent prove who it is — and stay within its lane? We'll answer it using two OAuth 2.0 grant types.

Grant Type 1

Auth Code + PKCE

How the AI Agent (Copilot) securely obtains a user-delegated access token from Keycloak — without embedding any client secret.

Grant Type 2

Token Exchange — RFC 8693

How the MCP Server narrows the agent's token scope before calling a Resource Server — enforcing least-privilege at every boundary.

Foundation: What MCP Protocol is, OAuth 2.1 vs 2.0, and why AI agents need their own identity.

Live Demo: GitHub Copilot reserves a conference ticket — using both grant types end-to-end.

Problem Statement

AI Assistants Have Evolved

The New Reality

  • Agents don't just chat anymore — they execute actions (write DBs, trigger pipelines).
  • If an AI calls an API without an identity, it's essentially an unauthenticated bot.
The core issue:
If your database cannot distinguish Copilot from a real human, you have a security void.

Common Pitfalls

God Mode: Giving the AI full API access via a single hardcoded API key.

Token Leakage: Naively forwarding the AI's token directly to the backend database.

Over-privilege: The AI can delete records, even if the human asking cannot.

Context Layer

Model Context Protocol (MCP)

MCP is the standard "USB-C port" linking AI models to external systems.

Build a server once, connect it to Claude, Copilot, Cursor, etc.

1. Tools

Functions the AI can execute. (e.g., reserveTicket)

2. Resources

Data the AI can read. (e.g., file://logs)

The Security Gap: MCP handles transport (JSON-RPC over HTTP/Stdio), but it does not enforce Auth out of the box. You must bring your own OAuth.

Evolution

OAuth 2.1 vs OAuth 2.0

OAuth 2.1 is essentially OAuth 2.0 with all the modern security best practices made mandatory.

Feature OAuth 2.0 OAuth 2.1
PKCE Optional / Recommended Mandatory for public clients
Implicit Flow Allowed (Prone to leaks) Removed entirely
Password Grant Allowed Removed entirely
Redirect URIs Wildcards allowed Exact string match only

Grant Type 2

Why perform a Token Exchange?

The Audience Problem

  • The Agent's token belongs strictly to the MCP layer (aud: mcp-server).
  • The Resource Server requires a database-level API token (aud: resource-api).
  • Forwarding the exact same token blindly violates zero-trust principles.

RFC 8693 Solution

1
Send Subject Token

MCP hands Keycloak the agent's token securely.

2
Keycloak Validates & Swaps

Issues a new token specifically sized for the backend API.

Deep Dive

Agent Token vs. Exchanged Token

① Agent Token  aud: mcp-inventory

Issued to GitHub Copilot via Auth Code + PKCE.

// Key claims
"iss": "https://localhost:9443/realms/oauth2-playbook",
"aud": "oauth2-playbook-mcp-inventory",   ← MCP audience
"sub": "6c681ad3-0d08-41c8-ad97-488921a4fa09",
"azp": "oauth2-playbook-mcp-agent",
"scope": "mcp:events:read
         mcp:reservations:write
         mcp:tickets:read
         openid profile",           ← mcp:* scopes
"preferred_username": "demo",
"name": "Demo User",
"exp": 1782167039,
"jti": "onrtac:e2237538-65ca-60da-9dcf-a64e84a087f1"

② Exchanged Token  aud: resource-server

Obtained by MCP Server via Token Exchange (RFC 8693).

// Key claims
"iss": "https://localhost:9443/realms/oauth2-playbook",
"aud": "oauth2-playbook-inventory-resource-server", ← new aud
"sub": "6c681ad3-0d08-41c8-ad97-488921a4fa09", ← same user!
"azp": "oauth2-playbook-mcp-inventory",
"scope": "api:inventory:reservations:write
         openid profile",           ← narrowed scope
"preferred_username": "demo",
"name": "Demo User",
"exp": 1782166790,
"jti": "ntrtte:5ec38678-4b6f-0195-6024-6f5e8aa60663"
Identity preserved, scope narrowed. Same sub, different aud & reduced scope — zero-trust boundary enforced.
